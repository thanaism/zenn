## DoS問題はどう解決されたか

前章までで、EntryPointによってガス精算の問題は解けたように見えます。
しかし、これだけではまだ足りません。

ERC-4337でbundlerは、複数のUserOperationを引数にしてEntryPointの`handleOps()`メソッドを呼び出します。そしてメソッドの中では、UserOperationが順に処理されます。

`handleOps()`は、検証のvalidation loopと実行のexecution loopに大きく分かれていました。これは、検証と実行を分離すると同時に、検証段階で先に支払い保証を確定するための構成です。

validation loopでは、ウォレットの`validateUserOp`で署名やnonceを検証し、paymasterがいる場合は`validatePaymasterUserOp`で支払い承認も検証します。そのうえでEntryPoint内のdepositから最大ガス代を払えるかを確認します。つまりこの段階では、「このUserOperationを実行してよいか」だけでなく、「失敗しても誰からガス代を回収できるか」も確定させています。

つまり、validation loopでrevertするUserOperationを流してしまうと、bundlerはガスを回収できなくなります。

一方で、validationを通ったあとの`execute`がrevertするのは別です。この段階では、EntryPointがすでに最大ガス代をdepositから確保しています。実行内容が失敗しても、EntryPointはそのUserOperationを失敗として扱い、消費したガスをsenderまたはpaymasterのdepositから回収してbundlerへ支払えます。

つまり、bundlerにとって危険なのはexecutionの失敗ではなく、validationを通らないUserOperationを掴まされることです。validation失敗はガスの払い損となり、これはDoS攻撃への脆弱性そのものです。
この事態を避けるために、**bundlerはEntryPointに送るuserOperationを選別する段階で、事前検証を行う必要が生じます**。

この検証を確実なものとするために、実は**validationフェーズでは使えるopcodeが制限されています**。
分かりやすいものとして時間依存の判定などは、事前検証時にOKでも、実際の実行時にNGとなってしまうパターンがありえますね。
したがって、他コントラクトの状態やブロック環境に依存する操作、つまりシミュレーション時点の結果が本番で覆りうる操作は使用できません。

そうして初めてシミュレーション結果が信用でき、bundlerは実行前に通る／通らないを確定できます。
ちなみに、このレギュレーションを標準化したのが、**ERC-7562**です。

validationとexecutionの分離は、AAの思想であると同時に、bundlerをDoSから守るための運用上の境界でもあるわけですね。

### 有効期限のあるSession Keyはどう実現しているのか

ここまで聞いて、勘の良い方は疑問に思ったかもしれません。

「え、せっかく検証ロジックを任意化したのに時間依存のコードはNGなの？てことは絶対に必要になりそうな『有効期間を指定した認可』すらできないの？」

安心してください。実際には、有効期限付きのSession Keyというのが普通に存在し、利用されています。

validationで有効期限を判定するとして、愚直な方法を考えるのなら、コントラクトウォレットの`validateUserOp`内で`block.timestamp`を読んで、現在時刻が期限内かどうかを確認すればよさそうに見えます。しかし、これは禁じられたopcodeを使用するため、先ほど述べたとおりNGな実装です。

そこでERC-4337では、**validationが「いま有効か」を直接判定するだけでなく、「このUserOperationがいつからいつまで有効か」をEntryPointとbundlerに返せるようになっています**。

具体的には、`validateUserOp`の戻り値である`validationData`に、`validAfter`と`validUntil`という時間範囲を返せるようになっています。

```solidity
interface IAccount {
    function validateUserOp(
        PackedUserOperation calldata userOp,
        bytes32 userOpHash,
        uint256 missingAccountFunds
    ) external returns (uint256 validationData);
}
```

`validationData`は、ざっくり言うと次の情報をpackedした値です。

```text
validationData = authorizer | validUntil | validAfter
```

`validAfter`は「この時刻以降なら有効」、`validUntil`は「この時刻までは有効」を表します。
つまり、ウォレットはvalidation中に現在時刻を読むのではなく、検証結果として有効期間を返します。

Session Keyの場合も同じです。

たとえばウォレットが「このsession keyは2026年6月1日から2026年6月30日まで、特定のtargetとfunctionだけ呼べる」というポリシーを持っているとします。
`validateUserOp`では、session keyの署名が正しいか、呼び出し先や関数が許可範囲内かを検証します。それらは同期的に検証して問題ないからです。

一方で、時間依存の判定は内部で行えないため、戻り値として返すという方法を取ります。
具体的には、session keyの有効期限を`validationData`の`validAfter`/`validUntil`として返します。

```solidity
function validateUserOp(userOp, userOpHash, missingFunds)
    external
    returns (uint256 validationData)
{
    SessionPolicy policy = decodeSessionPolicy(userOp.signature);

    require(isAllowedTarget(userOp.callData, policy));
    require(isAllowedFunction(userOp.callData, policy));
    require(recoverSigner(userOpHash, policy.signature) == policy.sessionKey);

    return packValidationData({
        authorizer: address(0),
        validAfter: policy.validAfter,
        validUntil: policy.validUntil
    });
}
```

こうすると、時間判定の責務はウォレットの任意コードから外れます。
bundlerはシミュレーション時にこの`validationData`を読み取り、「いまbundleに入れてよいか」「mempoolに置いておくならいつまで有効か」を判断できます。EntryPointもオンチェーン実行時に同じ有効期間を見て、期限外なら処理を通しません。

DoS対策と時間依存の条件を両立させるために、validation中の`block.timestamp`を参照する処理ではなく、**validation結果に有効期間のメタデータを返す**形で扱っているのがERC-4337の賢い方法です。
bundlerのシミュレーション結果を単にその瞬間のOK/NGという判定に落とすのではなく、「OKの場合、この時間範囲で有効」という期間を指定した判定結果として返しているわけですね。

### ERC-4337のまとめ

従来、ユーザーはEOAの秘密鍵で署名し、自らガスを支払ってmempoolにtxを送信していました。

一方で、ERC-4337ではtxの代わりにUserOperationを作成します。その際、コントラクトウォレットにある任意の検証コードに対応する署名形式を用いることができます（便宜上、署名と書きましたが任意コードの検証が通過できるなら署名という形式に限りません）。

このとき、第三者にガス代を支払ってもらいたい場合は、事前に代理支払者であるPaymasterコントラクトの検証ロジックに応じた支払許可証明データをUserOperationに含めておきます。

そのUserOperationをmempoolに送信する代わりに、代理送信者のbundlerが管理するalt-mempoolに送信することで処理を実行させます。

bundlerは、ガスの払い戻しを保証するEntryPointコントラクトを経由して、コントラクトウォレットの検証メソッドを実行させます。

UserOperationにPaymasterの支払許可データがある場合は、払い戻しはPaymasterが負担します。

すべてのvalidationフェーズが完了すると、UserOperationに指定された処理内容が無事、実行されます。
そして、**その際、処理内容の実行主体（msg.sender）はコントラクトウォレットのアドレスになります**（`wallet.execute()`メソッドで実行されるため）。

上記のほかに、userOperationで指定されたコントラクトウォレットが未デプロイだった場合に、実行前にデプロイも自動で行うためのinitCodeというプロパティなど、説明していない部分はいくつかありますが、ERC-4337がどうして今のような構成になっているのかを理解するために必要な説明はほぼ全てできたかと思います。
