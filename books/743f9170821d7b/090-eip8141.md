## EIP-8141によるネイティブAA構想

EIP-7702を超え、ネイティブAA構想として今まさに現在進行形で議論されているのがEIP-8141（Frame Transactions）です。

EIP-8141は、新しくタイプ6トランザクションを追加する提案です。
その内容を理解するために、まずは通常用途で広く使われているEIP-1559のタイプ2トランザクションを見てみましょう。

実際には細かいプロパティはたくさんありますが、タイプ2はざっくり以下のような構造でした。

```text
0x02 || rlp([
  nonce,
  destination,
  amount,
  data,
  signature
])
```

`nonce`で重複防止しつつ、`destination`、`amount`、`data`で宛先・金額・引数を指定します。
そして`signature`は、EOAの署名ですね。

これまで本書で説明してきたとおり、プロトコルが署名、nonce、balanceを固定ルールで検証する構成そのものです。
また、その実行内容は`destination`、`amount`、`data`で単一のものとなっています。

EIP-8141では、`destination`、`amount`、`data`という単一の実行先を`frames`という列に置き換えます。

```text
0x06 || rlp([
  nonce,
  sender,
  frames,
  signatures
])

frames = [
  [mode, target, value, data, ...],
  ...
]
```

ここで重要なのは、実行内容がtx直下の`destination`、`amount`、`data`ではなく、複数のframeに分解されていることです。
各frameは「どのmodeで、どのtargetに、どれだけのgasとvalueを渡し、どんなdataで実行するか」を持ちます。

現行draftでは、frameのmodeとして`DEFAULT`、`VERIFY`、`SENDER`が定義されています。

- `VERIFY`: 署名検証や支払い認可を行う
- `SENDER`: ユーザーの意図した処理を`sender`の文脈で実行する
- `DEFAULT`: デプロイやpost-opのような、`sender`そのものの実行ではない補助処理を行う

一番重要なのは`VERIFY`です。
`VERIFY` frameでは、アカウントやpaymasterが署名検証や支払い認可を行い、`APPROVE` opcodeによって「実行してよい」「支払ってよい」というtxスコープの承認を立てます。
その承認が立ったあと、`SENDER` frameで実際の処理を実行します。

たとえば単純な送金やコントラクト呼び出しなら、イメージはこうです。

```text
frame 0: VERIFY  senderの署名を検証し、実行と支払いを承認する
frame 1: SENDER  senderとしてtargetをcallする
```

ERC-4337で言えば、`validateUserOp()`に相当する処理が`VERIFY` frameへ、`execute()`に相当する処理が`SENDER` frameへ移ったと見ると分かりやすいです。
Paymasterの検証も、別の`VERIFY` frameとしてtxの中に置けます。

これにより、検証・支払い・実行という、ERC-4337が概念的に分離していた処理を、プロトコルネイティブなトランザクション構造そのものとして表現できます。
EntryPointという外部コントラクトに処理順序を握らせるのではなく、txそのものに「検証して、支払いを確定して、実行する」という順序を持たせるわけです。

### default code

さらに特筆すべきは`default code`という仕組みです。
EIP-8141では、`VERIFY` frameにEOAが指定されたときに、`default code`というフォールバック処理が走るようになっています。
`VERIFY` frameでEOAが呼ばれると、tx内の`signatures`にそのEOAの署名が含まれているかを確認し、含まれていれば`APPROVE`によって実行や支払いを承認します。
つまり、EIP-7702のdelegation indicatorの指定も不要、新しいコントラクトウォレットへの移行も不要でVERIFYを利用できます。ERC-4337のbundlerも要りません。

EIP-7702では、ユーザーがdelegationを明示的に行う必要がありました。
delegationを誰が設定できるかはウォレット実装に依存するので、ウォレットプロバイダーがdapp向けにその操作を解放しなければ、ユーザーは結局ウォレット側の判断待ちになります。
EIP-8141はそういったしがらみもなく、まさにネイティブAAと言えるでしょう。

### DoS問題

一方で、DoS問題は、この提案でも避けて通れません。
EIP-8141の検証は`VERIFY`フレームの中で任意コードを走らせる以上、ERC-4337と同じく、mempool段階で重い検証だけをノードに実行させる攻撃にさらされます。

そこで出てくるのがvalidation prefixです。
これは、支払者が確定するまでに実行される最短のframe列を指します。

public mempoolに載せる前に、このvalidation prefixをどこまで検証するか。
そこにどんなopcode・storage access制限を課すか。
これがEIP-8141の中心的な論点になります。

これはERC-4337で使われたERC-7562系の考え方[^1]を、frame transactionの文脈へ移す話です。
最初のEIP-86以来EOAを縛り続けてきた制約が、10年後の最先端の提案でもまだ核心に居座っているわけです。

[^1]: 以前の章でも説明したもの。validationフェーズで使用してよいopecodeを制限する。

### EIP-8141のステータス

EIP-8141は、EIP-86が掲げた「全アカウントをコントラクトに、検証を自由に、プロトコルに刻む」という原初のビジョンへ10年かけて戻ってきた提案だと言えます。

なお、EIP-8141は議論中の提案であり、このままEthereumに採用されると決まったわけではありません。
2026年前半時点では、2026年後半に予定されるHegotáアップグレードに向けてCFI（Considered for Inclusion）というステータスにあり、確定したheadlinerではありません。同じくネイティブAAを狙う競合提案も複数あります。「ネイティブAAをプロトコルに載せる」という方向自体はほぼ既定路線ですが、それがEIP-8141という形で着地するかはまだコア開発者間の収束待ち、というのが正確なところです。
